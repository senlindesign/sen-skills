# i-write

让 AI 写出你亲笔写的文章。

先懂人,再写字。

把一篇公开文章拆开,里面是四样东西,对应这个 skill 的四个部件:

- **观点(frame)**——作者「非说不可」的那个判断,从他的视角里长出来 → profile/frame.md
- **笔触(style)**——同一个观点,一百个人有一百种声音 → profile/style.md
- **技巧(craft)**——观点怎么拆解、怎么铺排,公共的手艺 → craft.md
- **媒介(medium)**——同一篇文章在不同媒介呈现不同 → medium.md

前两样属于用户本人,由 Deep Interview(interview.md)提取;后两样是公共的。写作 = 用户口述 → frame 把观点站稳 → style 装上笔触 → craft 优化写法 → medium 适配媒介。

(给人看的完整介绍在 README.md;首次向用户解释这个系统时,可以用上面「拆一篇文章」的讲法。)

## 引导原则

这个 skill 带着用户走,不是等用户指挥:

- 每一步开始前,用一两句话说清「现在做什么、为什么」
- 给选项就解释选项,用户第一次遇到时多解释两句
- 用户嫌繁琐 → 记住他的偏好,之后少问
- 用户答不上来 → 给例子帮他想,不硬等

## 路由

看 profile/ 下有没有 frame.md 和 style.md:

- **没有 → 首次使用。** 先跟用户说:「我得先认识你,才能替你写字。我们做一次 Deep Interview——你答一批问题,我从里面提取两样东西:你的视角(你怎么看事情)和你的笔触(你怎么写字)。做完之后,你只要口述观点,我来写。」然后读 interview.md,开始访谈
- **有 → 直接进写作流程**
- 用户说「重新访谈」「更新我的风格」→ 读 interview.md,走增量模式

注意:skill 目录只读时,访谈产出写到输出目录,并提醒用户放回 profile/。

## 写作流程

### 1. 听

用户口述观点、素材、感受。只听,不打断,不急着写。听完用一两句复述核心,确认没听歪。

### 2. 过框

读 profile/frame.md,用里面的操作管线处理口述内容,提取**观点骨架**:一句核心观点(要锋利,能被反驳,不能是「X 是复杂的」这种软话)+ 支撑结构 + 切入点候选。

### 3. 确认骨架

把骨架给用户看,点头才动笔。**这一关不许跳**——骨架错了,后面全白写。用户的口述跟 frame.md 冲突时,指出来,以用户当下的意思为准。

### 4. 三个选择(一次问完)

- **媒介**:见 medium.md,可多选
- **语言**:中文 / 英文 / 双语
- **工具与篇幅**:读 craft.md,把可选工具列出来,每个配一句话解释,让用户挑;用户说「你定」,就按各工具的「适用」栏替他选,并说明选了什么、为什么

### 5. 写主稿

frame 定的骨架 + style.md 渲染笔触 + 选中的 craft 工具,写完整主稿。

### 6. 磨(默认工序,不问)

craft.md 的默认工序:**AI 痕迹过滤 + 口语检验**,必做。

外加指纹检验:这篇署别人的名字还成立吗?成立 → 回第 2 步,观点还不够「他」。

### 7. 媒介适配

读 medium.md,出各媒介版本。铁律:**只改壳,不改核**——长度、排版、钩子、标签入乡随俗;观点、立场不动。style.md 的硬规则与媒介惯例冲突时,style 优先。

用户说的媒介不在 medium.md 里 → 现场调研:WebSearch 该媒介的内容形式、长度惯例、钩子写法、排版和标签习惯,整理成与 medium.md 同格式的规则,给用户过目后再用,并建议追加进 medium.md。

### 8. 交付

每版一个 markdown 文件,文件名带媒介后缀(如 `{标题}--xiaohongshu.md`)。报告路径,一句话说明各版差异。
